# Board — кросс-проектная доска

Ты ведёшь единый Kanboard-дашборд планов по всем проектам секретаря. Доска — **проекция**:
канон планов живёт в самих проектах (git/Trello/файлы), доска — read-model поверх них.

**PoC — односторонний.** Только проекты → доска. Доска ридонли: комментарии с карточек не
читаешь и в проекты ничего не пишешь. Обратный поток (замечания → бэклоги) добавим позже.

Раскладка Kanboard: **один board, swimlane на проект, колонки `Идеи` · `To Do` · `Готово`.**
Колонки «В работе» нет — что крутится прямо сейчас, видно по статусам воркспейсов Orca, на
планировочной доске это не дублируем.

Пишешь на доску не через сырой API, а через детерминированные подкоманды плагина (см. «Процедура»).
Всё суждение — что тянуть, в какую корзину, на какой высоте — твоё; подкоманды только исполняют.

## Что откуда берёшь

- **Список проектов и где у каждого план** — `~/control-panel/docs/projects.md` (таблица
  «Планировочный источник» + абзац нюансов). Оттуда узнаёшь, что сканировать и как: git-репо с
  `backlog.md`, Trello-доска, `findings/*.md`, корень без git и т.п. **Все проектные особенности —
  там, а не в этом скилле.** Скилл общий; частности читаешь из манифеста. Список проектов, у которых
  есть план (те, кому заведён swimlane), отдаёт `board projects`.
- **Планы ты собираешь руками.** Питон их не парсит (планы разнородны, жёсткий парсер хрупок).
  По каждому проекту сам: `git -C ~/projects/<repo> pull`, прочитай плановые файлы из манифеста,
  для `inspect_notebooks` — Trello по кредам из его `.env`. Разложи по корзинам своим суждением.
- **Kanboard** — креды в `~/control-panel/.env` (`KANBOARD_URL`, `KANBOARD_API_USER`,
  `KANBOARD_API_TOKEN`). Orca env не прокидывает, поэтому **сорсишь их сам** перед каждым вызовом
  плагина: `set -a && . ~/control-panel/.env && set +a`. Дальше через них работают подкоманды.

## Корзины (общие для всех проектов)

План любого проекта раскладываешь по четырём корзинам. Если корзины у проекта нет — оставляешь
пустой, ничего не выдумываешь.

- **Идеи** — future/icebox, не запланированная работа → колонка `Идеи`.
- **Открытые** — задачи в работе или к работе (TODO/OPEN/PARTIALLY и аналоги) → колонка `To Do`.
- **Готово** — закрытое. На доску тянешь **только последние 5 по дате** → колонка `Готово`.
  Остальное закрытое остаётся в архиве плана проекта, на доску не идёт (доска = текущий вид, план
  проекта = полный журнал).
- **Не делаем** — obsolete/сознательно отклонённое. **На доску не тянешь вообще** (это анти-задачи,
  чтобы не всплывали снова).

Проекты со структурой «Открытые/Идеи/Не делаем/Архив» (как `service-template/docs/backlog.md`)
ложатся тривиально. Свободный план раскладываешь по тем же корзинам сам, по смыслу.

## Высота: доска — дашборд, не зеркало

Не взрывай каждый лист плана в карточку. Карточки на высоте «текущий фокус»: что в работе и что
дальше, а не весь резервуар задач. Если у проекта есть свой сводный слой («где мы сейчас» —
`STATUS.md` и т.п.) — читай его; большой бэклог-резервуар остаётся в плане проекта, на доску идут
только текущий срез и пара «next». У спринтовых проектов высота карточки — спринт/фаза, не пункт
бэклога. Внутренняя машинерия планирования (пайплайны, воркфлоу-скиллы проекта) — не планировочный
контент, на доску не идёт. Сколько и с какой высоты тянуть у конкретного проекта — в `projects.md`.

## Процедура

Все подкоманды — из своего воркспейса (стартовый cwd — воркстри board-агента), выводят JSON.
Перед каждым bash-вызовом плагина подгружай креды: `set -a && . ~/control-panel/.env && set +a`.

**Область прогона.** Если тебе передан аргумент-проект (например `/board inspect_notebooks`) —
синкай ТОЛЬКО его swimlane: бери список через `board projects --only <name>`, дальше шаги 2-3 для
одного проекта, остальные не трогай. `setup` всё равно прогони (идемпотентен). Без аргумента —
все проекты (`board projects`).

### 1. Убедиться в структуре доски

```
python3 -m triggered_agents board setup
```

Идемпотентно: создаёт board-проект, приводит колонки к `Идеи/To Do/Готово`, заводит swimlane на
каждый проект из `projects.md`. Гонять в начале каждого прогона безопасно.

### 2. Собрать планы

`python3 -m triggered_agents board projects` — список проектов с планом (по одному swimlane на
каждый). Для каждого сам подтяни и прочитай план (git pull + плановые файлы / Trello, как указано в
`projects.md`). Разложи по корзинам и выбери высоту (см. выше).

### 3. Свести план на доску

Для каждого проекта (его swimlane):

1. `python3 -m triggered_agents board cards --swimlane <проект>` — текущие карточки этого swimlane.
2. Сопоставь план с карточками по полю **`reference`** — стабильный ключ пункта плана. Ключ
   присваиваешь при первом появлении пункта и **дальше не меняешь**; удобно префиксом проекта
   (`dnd:sprint-007`, `orca:pr-ideas`). Матчь к существующим, не плоди дубли на переформулировках.
3. Приведи swimlane к плану:
   - каждый пункт → `board upsert --swimlane <проект> --ref <ключ> --column <Идеи|To Do|Готово>
     --title <...> [--description <...>]` (создаст новую карточку или обновит+переместит по `ref`);
   - карточки, которых в плане больше нет → `board archive --ref <ключ>`.
4. Доска — проекция: расхождение текста/колонки перезатираешь под план.
5. **Дрейф план↔реальность.** Глянь свежие коммиты репо (`git -C ~/projects/<repo> log`). Если в
   репо есть закрытая работа, которой нет в плане (план заморожен, работа шла мимо спринта) — заведи
   для неё карточки в `Готово` прямо из реальных коммитов, с пометкой «вне спринта» и своим `ref`.

## Инварианты

- **Не задавай уточняющих вопросов.** Headless-прогон без человека — вопрос повесит сессию.
  Действуй по лучшему суждению; сомнительное классифицируй консервативно.
- **Доска — проекция, планы — канон.** PoC односторонний: в репозитории проектов НИЧЕГО не
  коммитишь, только читаешь. Единственная запись — на доску, через подкоманды.
- **`reference` стабилен**, не меняется на переформулировках — иначе доска плодит дубли.
- **`Готово` — максимум 5 карточек на проект.** Не тащи весь архив.
- **Проектные особенности — в `projects.md`, не здесь.** Скилл общий для всех проектов.
- Пиши на русском, кратко, без AI-штампов (без em-dash ради драмы, без «стоит отметить»).
