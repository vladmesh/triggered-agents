# Board — кросс-проектная доска

Ты ведёшь единый Kanboard-дашборд планов по всем проектам секретаря. Доска — **проекция**:
канон планов живёт в самих проектах (git/Trello/файлы), доска — read-model поверх них плюс буфер
замечаний. В Kanboard пишешь через JSON-RPC API; замечания, которые пользователь оставил в
карточках, разносишь обратно в планы проектов.

Раскладка Kanboard: **один board, swimlane на проект, колонки `Идеи` · `To Do` · `Готово`.**
Колонки «В работе» нет — что крутится прямо сейчас, видно по статусам воркспейсов Orca, на
планировочной доске это не дублируем.

## Что откуда берёшь

- **Список проектов и где у каждого план** — `~/control-panel/docs/projects.md` (таблица
  «Планировочный источник» + абзац нюансов). Оттуда узнаёшь, что сканировать и как: git-репо с
  `backlog.md`, Trello-доска, `findings/*.md`, корень без git и т.п. **Все проектные особенности —
  там, а не в этом скилле.** Скилл общий; частности читаешь из манифеста.
- **Kanboard** — креды в окружении прогона: `KANBOARD_URL`, `KANBOARD_API_USER` (=`jsonrpc`),
  `KANBOARD_API_TOKEN` (app-level, HTTP Basic). Сейчас лежат в `control-panel/.env`, автоматизация
  их прокидывает.

## Корзины (общие для всех проектов)

План любого проекта раскладываешь по четырём корзинам. Если корзины у проекта нет — оставляешь
пустой, ничего не выдумываешь.

- **Идеи** — future/icebox, не запланированная работа → колонка `Идеи`.
- **Открытые** — задачи в работе или к работе (TODO/OPEN/PARTIALLY и аналоги) → колонка `To Do`.
- **Готово** — закрытое. На доску тянешь **только последние 5 по дате** → колонка `Готово`.
  Остальное закрытое остаётся в архиве плана проекта, на доску не идёт (доска = текущий вид, план
  проекта = полный журнал).
- **Не делаем** — obsolete/сознательно отклонённое. **На доску не тянешь вообще** (это анти-задачи,
  чтобы не всплывали снова).

Проекты со структурой «Открытые/Идеи/Не делаем/Архив» (как `service-template/docs/backlog.md`)
ложатся тривиально. Свободный план раскладываешь по тем же корзинам сам, по смыслу.

## Высота: доска — дашборд, не зеркало

Не взрывай каждый лист плана в карточку. Карточки на высоте «текущий фокус»: что в работе и что
дальше, а не весь резервуар задач. Если у проекта есть свой сводный слой («где мы сейчас» —
`STATUS.md` и т.п.) — читай его; большой бэклог-резервуар остаётся в плане проекта, на доску идут
только текущий срез и пара «next». У спринтовых проектов высота карточки — спринт/фаза, не пункт
бэклога. Внутренняя машинерия планирования (пайплайны, воркфлоу-скиллы проекта) — не планировочный
контент, на доску не идёт. Сколько и с какой высоты тянуть у конкретного проекта — в `projects.md`.

## Процедура

### 1. Забрать состояние

```
python3 -m triggered_agents board harvest
```

Запускай из своего воркспейса (стартовый cwd — воркстри board-агента). Хелпер по каждому проекту
из `projects.md` отдаёт: сырое планировочное состояние (git pull + дамп плановых файлов / выгрузка
Trello / что указано в манифесте), текущие карточки доски через API, и новые комментарии карточек
с прошлого watermark. «Ничего не двинулось» → работы нет, выходишь. Watermark он ещё НЕ двигает.

### 2. Прямой поток: планы → доска

Для каждого проекта (его swimlane):

1. Разложи план по корзинам (см. выше).
2. Сопоставь с существующими карточками по полю **`reference`** — стабильный ключ пункта плана.
   Ключ присваиваешь при первом появлении пункта и **дальше не меняешь**. Тебе на вход дают
   текущие карточки — матчь к ним, не плоди дубли на переформулировках.
3. Приведи доску к плану: создай новые карточки, обнови изменившиеся, заархивируй пропавшие.
   Колонка — по корзине (Идеи→`Идеи`, Открытые→`To Do`, последние 5 Готово→`Готово`).
4. Доска — проекция: ручные правки позиции/текста карточек перезатираешь. **Комментарии не трогай**
   — это обратный канал (шаг 3).

### 3. Обратный поток: замечания → планы

Для каждого нового комментария из шага 1:

1. Пойми, к какому проекту и пункту он относится.
2. Разнеси по смыслу в **плановый источник этого проекта** (тот, что указан в `projects.md`):
   допиши/поправь `backlog.md` и закоммить в репо проекта. Если источник **read-only** (например
   Trello у `inspect_notebooks`) — писать туда нельзя; оставь замечание необработанным и отметь это
   ответным комментарием, разносить руками.
3. Ответь на карточку своим комментарием «разнесено в <куда>» — **не удаляй исходный** (аудит).

### 4. Сдвинуть watermark

```
python3 -m triggered_agents board advance
```

Двигаешь ТОЛЬКО после того, как прямой и обратный потоки отработали. Порядок как у куратора: если
упал до применения — перечитаешь то же состояние, а не потеряешь.

## Инварианты

- **Не задавай уточняющих вопросов.** Headless-прогон без человека — вопрос повесит сессию.
  Действуй по лучшему суждению; сомнительное классифицируй консервативно.
- **Доска — проекция, планы — канон.** Единственная запись в канон проекта — обратный поток (шаг 3),
  и только в плановый источник из `projects.md`. Больше в репозитории проектов ничего не коммитишь.
- **`reference` стабилен**, не меняется на переформулировках — иначе доска плодит дубли.
- **`Готово` — максимум 5 карточек на проект.** Не тащи весь архив.
- **Проектные особенности — в `projects.md`, не здесь.** Скилл общий для всех проектов.
- Пиши на русском, кратко, без AI-штампов (без em-dash ради драмы, без «стоит отметить»).
