# Оценка Beads как замены Kanboard под доску Pipeline

Карточка: `triggered-agents-242`. Дата: 2026-07-03. Методика — та же, что в
`docs/agent-kanban-eval.md` (branch `docs/agent-kanban-eval`, ещё не в main): таблица
10 требований пайплайна к доске, колонка на каждого кандидата.

Кандидаты: [gastownhall/beads](https://github.com/gastownhall/beads) (`bd`, Go, Dolt-движок,
MIT) и его порт [Dicklesworthstone/beads_rust](https://github.com/Dicklesworthstone/beads_rust)
(`br`, Rust, SQLite+JSONL, MIT+rider). Оба протестированы живьём: скачаны прекомпилированные
Linux-бинарники релизов (`bd v1.0.4`, `br v0.2.16`), без Go/Rust тулчейна в окружении.

## Рекомендация

**Не мигрировать сейчас. Взять две вещи: атомарный claim и нативные deps/ready-detection —
как референс при следующей правке `ops.py`/`claim_card`, не как повод переезжать.** Если всё же
решим переезжать — **оригинал `bd`, не `br`.**

Коротко почему: обе главные фичи, ради которых смотрели на beads, подтвердились живой пробой —
`--claim` атомарен (проверено гонкой двух процессов, 10/10 попыток ровно один победитель), deps
и `ready`/`blocked` работают корректно из коробки. Это первый кандидат в этой серии оценок
(после agent-kanban), который реально закрывает claim-гонку нативно, а не эмуляцией. Но у нас
эта гонка уже закрыта host-lock'ом (`STATE.lock()` в `claim_card`) и она ни разу не была
источником инцидентов — переезд купил бы страховку от риска, которого пока не было. А по
остальным пунктам, которые для нас реальны сегодня, — метаданные, 6 колонок, комментарии — либо
паритет, либо доработка не меньше той, что требовалась agent-kanban. Плюс главный архитектурный
зазор: `bd`/`br` по умолчанию — embedded движок на локальном файле, а Kanboard сегодня — сетевой
JSON-RPC сервер (`KANBOARD_URL`), к которому дispatcher/worker/reviewer ходят как клиенты
независимо от хоста. Чтобы не потерять это, нужен `dolt sql-server`/`--shared-server` —
дополнительный сервис, который надо поднимать и обслуживать, ровно то же бремя, что несёт
сегодняшний Kanboard.

## Живая проба

Окружение: нет Go/Rust/Docker, есть sudo и интернет. `bd`/`br` — статические Go/Rust бинарники,
собраны без тулчейна.

```
$ curl -sL -o bd.tar.gz .../beads_1.0.4_linux_amd64.tar.gz && tar xzf bd.tar.gz
$ ./bd --version
bd version 1.0.4 (ce242a879)
$ curl -sL -o br.tar.gz .../br-0.2.16-linux_amd64.tar.gz && tar xzf br.tar.gz
$ ./br --version
br 0.2.16
```

Цикл create → deps → ready-detection → close, на `bd` (в пустом git-репо, `bd init --prefix demo`):

```
$ bd create "Predecessor task" -t task --json          # -> demo-xdd
$ bd create "Successor task, blocked on predecessor"    # -> demo-s65
$ bd dep demo-xdd --blocks demo-s65
{"blocked_id":"demo-s65","blocker_id":"demo-xdd","status":"added","type":"blocks"}

$ bd ready --json      # только demo-xdd (предшественник, без блокеров)
$ bd blocked --json    # demo-s65, blocked_by: [demo-xdd]

$ bd close demo-xdd --reason "done"
$ bd ready --json       # теперь demo-s65 — ready-detection пересчитался сам
```

Конкурентный claim — два фоновых процесса, `--claim` (= "assignee = actor, status = in_progress,
только если ещё не занята") на одну и ту же карточку одновременно, 6 попыток на `bd` + 4 на `br`:

```
$ ( bd update demo-0ae --claim --actor workerA --json > A.json 2> A.err ) &
$ ( bd update demo-0ae --claim --actor workerB --json > B.json 2> B.err ) &
$ wait
A exit 1   B exit 0
$ cat A.err
Error claiming demo-0ae: issue already claimed by workerB
$ bd show demo-0ae --json | jq '.assignee, .status'
"workerB"  "in_progress"
```

10 запусков подряд (6 на `bd`, 4 на `br`, свежая карточка каждый раз) — **ровно один победитель
каждый раз**, ровно один ненулевой exit code с внятной ошибкой у проигравшего, никогда оба не
записались. `br --claim` ведёт себя идентично (proigravший получает exit 4). Никакого
дополнительного host-lock не потребовалось — атомарность нативная, на уровне самого движка
(Dolt/SQLite транзакция с условием), а не эмулируется CLI-обвязкой.

```
$ bd close demo-0ae demo-dj1 demo-47z ... --reason "test cycle done"
$ bd status
Total Issues: 8   Open: 0   In Progress: 0   Closed: 8   Ready to Work: 0
```

Полный цикл create → deps → ready → конкурентный claim → close прошёл на обоих кандидатах без
единой ручной правки движка.

## Что доска обязана уметь

10 требований, источник и метаданные — как в `docs/agent-kanban-eval.md` (`ops.py`, `model.py`
этого репо, не изменились). Кратко: атомарный claim, уникальный `reference`, 6 flat-метаданных
(`task_type, project, blocked_by, head, claim, slug`), 6 именованных колонок
(`Идеи, Ready, In progress, Validate, Blocked, Done`), позиция=приоритет, комментарии с ts,
список активных карточек, контейнер-проект через API, свимлейны (nice), title+description (nice).

## Сравнение по требованиям

| # | Требование | Kanboard сейчас | agent-kanban as-is | `bd` (beads) as-is | Вердикт |
|---|---|---|---|---|---|
| 1 | Атомарный claim | нет CAS; эмулируется host-lock + 1 диспетчер | нет CAS, слепой `UPDATE WHERE id` | **нативно**: `update --claim` = CAS (assignee IS NULL), подтверждено гонкой 10/10 | ✔ лучше |
| 2 | reference, поиск по нему | поле `reference`, `getTaskByReference` | id = `KEY-NNN`, клиент задаёт | `--id demo-triggered-agents-242` при create, `bd show <id>` — id и есть reference, первичный ключ | ✔ лучше (не вторичное поле, а сам PK) |
| 3 | Метаданные (flat str→str, 6 ключей) | `saveTaskMetadata` | нет (жёсткая схема) | `--set-metadata k=v` / `--metadata '{"a":"b"}'` при create, произвольные ключи, `--metadata-field k=v` фильтр в `list` | ✔ паритет |
| 4 | 6 именованных колонок, CRUD+move | реальные, API | статус — ENUM из 4 | статус — built-in enum (7 значений: open/in_progress/blocked/deferred/closed/pinned/hooked) + `status.custom` для доп. состояний (проверено: добавил `idea`,`ready`,`validating` без конфликта) | ~ достижимо, но статус ≠ колонка с позицией, это FSM-labels |
| 5 | Позиция в колонке = приоритет | порядок в колонке | отдельный ENUM priority | `priority` 0-4 (`-p`/`P0-P4`), нет позиции-в-статусе как в Kanboard | ~ иначе (ordinal приоритет, не drag-order) |
| 6 | Комментарии: автор+ts, маркеры-канал | `createComment`/`getAllComments` | `POST/GET comments`, immutable | `bd comment <id> "text"` / `bd comments <id> --json`, поля `author, text, created_at`; свободный текст, маркер `[report:done]` как префикс проходит как обычная строка | ✔ паритет |
| 7 | Список активных карточек, open/closed | `getAllTasks(status_id)` | `GET /projects/:id/tasks` | `bd list` (default open), `bd ready`, `bd blocked`, `--all`, фильтры `--metadata-field`/`--label`/`--assignee` | ✔ лучше (готовые ready/blocked view, не только сырой список) |
| 8 | Контейнер-проект через API | `createProject` (живой RPC на бегущий сервер) | `POST /api/projects` | нет аналога: "проект" = `.beads` в git-репо, создаётся `bd init` один раз оффлайн, не рантайм-вызовом внутри работающей доски | ~ другая модель, см. ниже |
| 9 | Свимлейны (per-project) | нативные | группировка по epic в браузере, без модели | нет нативных; `project` как `--metadata-field`/label — фильтруется, но нет UI-группировки (вне скоупа: пайплайн ходит только через CLI) | ~ функционально заменимо, визуально нет |
| 10 | title + description | `description` | текст в теле | `--title`/`-d/--description` при create, `--body-file` | ✔ паритет |

## Центральная доска vs per-repo `.beads`

`bd`/`br` спроектированы вокруг «одна `.beads` на git-репо, агент работает внутри этого репо»
(GasTown-паттерн). Ничто, впрочем, не требует, чтобы `.beads` жила именно в репо кода проекта:
это может быть один выделенный репо-«доска» (как сегодня отдельный Kanboard-сервер, не привязанный
к репо ни одного из проектов), а `project`/swimlane-эквивалент — обычный metadata-ключ или label
на карточке, ровно как `META_PROJECT` сегодня. В этой конфигурации «одна доска, много
проектов-свимлейнов» переносится без структурных потерь — вопрос не в том, можно ли централизовать,
а в том, как именно.

Настоящий зазор — не свимлейны, а транспорт. Embedded-режим (`bd init`, дефолт) — это локальный
файл/Dolt-каталог: конкурентный доступ работает только с общей файловой системы (наш `claim`-тест
выше это и проверил — два процесса на одном хосте). Сегодняшний Kanboard — сетевой JSON-RPC
(`KANBOARD_URL`, HTTP Basic), к нему ходят по сети независимо от хоста воркера. Чтобы не потерять
это свойство, `bd` нужно поднимать в `--server`/`--shared-server` режиме (внешний
`dolt sql-server`, единая база `~/.beads/shared-server/` на несколько проектов) — то есть
эксплуатационно то же самое, чем Kanboard-сервер является уже сейчас: отдельный демон, которым
надо владеть. Готовое `--shared-server` — плюс (не пришлось бы городить своё), но это всё равно
новый сервис в инфраструктуре, а не «просто CLI без сервера», как могло показаться по описанию
beads в контексте задачи.

Комментарии/маркеры как канал управления (наш `[report:done]`/`[review:red]`/т.п.) переносятся
без изменений: `bd comment`/`bd comments --json` дают то же самое — свободный текст с автором и
таймстампом, маркер как договорной префикс, который парсит наш код, а не движок.

## Оригинал vs `br`

**`bd` (оригинал, Go+Dolt, GasTown/gastownhall) — очень активен.** 25k звёзд, последний коммит
на момент оценки — сегодня же (`b35502d`, merge PR #4525, 2026-07-03T17:14), несколько
коммитов в день (`fix(export): auto-export JSONL in SQL-server mode`, `feat(gate): turn the
smart remote-migrate gate on by default`). Полный CLI (`swarm`, `federation`, `gate`, `formula`,
азуре/гитлаб/notion-интеграции) говорит о быстро растущей площади API — риск, что то, что мы
захостим под себя, за полгода станет unrecognizable, реален.

**`br` — предобзор называл его «замороженным портом», это больше не так.** Живая проверка
(`git log`, GitHub API) на момент оценки: последний коммит — сегодня же, 2026-07-03
(`c3c8abfa`, "deps: bump rust-minor group"), с фичами в тот же день
(`f9d3fecf feat(show): soft-wrap...`, `2130ac25 feat(delete): purge all tombstones...`), 30
релизов, версия дошла до v0.2.16. Это активно поддерживаемый порт, синхронизирующийся с
апстримом `bd`, не замороженный снимок. Премиза брифинга устарела к моменту фактической
проверки — стоит поправить это в канонической записи, если она где-то фиксировалась.

Что это меняет для решения «оригинал vs `br`», раз оба движутся:

- **`bd`** — Dolt-движок (полноценная версионируемая SQL-БД, git-подобная история на уровне
  строк) — тяжелее эксплуатационно (свой бинарник Dolt внутри, `dolt sql-server` для
  сетевого режима), но даёт нативные `--shared-server`/`federation`/`repo` для мульти-проектных
  сценариев, которые нам и нужны для централизации. Метаданные (`--set-metadata`, произвольный
  JSON) — полный паритет с нашими 6 ключами.
- **`br`** — SQLite+JSONL, легче поставить и понять (один статический бинарник, знакомый sqlite
  файл), но **нет аналога произвольных key-value метаданных**: только labels (теги) и один
  непрозрачный `agent_context` JSON-блоб без фильтрации по ключу (в отличие от `bd`'s
  `--metadata-field k=v`). Три из наших шести ключей (`blocked_by` → нативный dep, `claim`/`head`
  можно притянуть к `assignee`/labels) легли бы нормально, но `task_type`/`project`/`slug`
  пришлось бы паковать в `agent_context` без индекса — писать свой фильтр поверх JSON-парсинга,
  то есть терять то единственное, что `bd` даёт бесплатно.
- И `bd`, и `br` — **движущаяся мишень**, оба меняются день в день. Для нашего `ops.py` это
  означает: любая миграция фиксирует нас на конкретной версии CLI-контракта (флаги `--claim`,
  формат `--json`, коды выхода), и апгрейды выше по семверу надо будет регресс-тестировать так
  же, как сегодняшние апгрейды Kanboard — разницы в риске "движущаяся цель" между `bd` и `br`
  на практике не оказалось, оба одинаково активны.

Если решим мигрировать — **`bd`**, не `br`: метаданные есть из коробки, `--shared-server`
закрывает архитектурный зазор централизации, а «легче/проще» у `br` не компенсирует потерю
произвольных метаданных, которые пайплайну нужны буквально (`task_type`, `slug`, `head`).

## Что взять к себе, даже если не мигрируем

- **Условный CAS-эндпоинт для claim** — `bd update --claim` буквально то, чем должен был бы стать
  `claim_card`, не будь Kanboard тем, что есть: `UPDATE ... WHERE assignee IS NULL`, конфликт —
  внятная ошибка с именем текущего владельца, не молчаливая перезапись. Стоит унести саму
  формулировку ошибки (`"already claimed by X"`) в `model.GuardError` вместо текущего
  `"already claimed by {meta[...]}"`, — там уже так и есть, но именно эта форма, подтверждённая
  живым тестом на независимом движке, — хороший аргумент не трогать `claim_card`'s host-lock,
  раз он уже даёт тот же гарантированный результат без сетевого движка под ним.
- **Нативный `ready`/`blocked` view поверх deps** — прямо то, что `claim_card` сегодня делает
  вручную (перебор `getAllTasks` + `getTaskMetadata` на предшественника). Не повод менять
  движок, но повод при следующей правке `claim_card` держать в голове эту форму API:
  «дай мне список без активных блокеров» одной командой, а не N+1 перебором.
- **`--metadata-field`/`--has-metadata-key` фильтр в списке** — Kanboard такого не даёт
  (`list_cards` в `ops.py` фильтрует уже после `getAllTasks`, в Python, а не на сервере);
  небольшой, но реальный выигрыш по числу RPC при росте доски.
- **`status.custom`** — способ добавить состояние без ALTER TYPE/миграции, который стоит держать
  в уме как референс, если 6-колоночный FSM когда-нибудь захочет вырасти.

## Риски

- **Незрелость под наш профиль нагрузки.** Оба движка обкатаны на одиночных агентских
  воркфлоу (`bd prime`, git-hooks, `AGENTS.md`), не на диспетчере с cap'ом и watchdog-ретраями.
  Дважды заявленная «нативная поддержка агентов» на практике означала «один агент, один claim»
  — ровно то, что мы и проверили; многоагентный сценарий с ретраями, per-project cap и
  сложной guard-цепочкой (`claim_card`, 60+ строк) остаётся целиком нашим кодом независимо от
  движка.
- **Эксплуатационный вес.** `bd`(Dolt) — новый бинарник-зависимость с собственным движком БД;
  `br`(SQLite) — легче, но без метаданных. Оба требуют `--server`/`--shared-server` для
  сетевого доступа, которого сегодня Kanboard даёт бесплатно.
- **Движущаяся мишень с обеих сторон** — см. «Оригинал vs `br`» выше: премиза "br заморожен"
  не подтвердилась, оба кандидата активно меняются, разница в риске между ними меньше, чем
  предполагалось до живой пробы.
- **Стоимость миграции** та же, что у agent-kanban: связанность локализована в `ops.py`
  (движок задуман подменяемым), но 6→N-статусов и position→priority — реальный порт модели,
  не смена транспорта.
